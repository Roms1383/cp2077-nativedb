Documentation guidelines

SYNTAX

Here are some examples on how you can write documentation to format it. On the left is what you write, on the right is how it will look like. Keep reading to the section DO & DON'T DO below to learn some writing styles.

NEWLINES

Welcome fellow modder! This is a newline.  This is a new paragraph :)

TYPE REFERENCES

You can reference types like [Vector4] and they are clickable. It works with primitive types like [Bool], [CName] but without redirection. If type is misspelled (case-sensitive) or not found, it is formatted but redirection will not work like [matRix].

SELF REFERENCES

Let's suppose you are documenting the struct [GameTime].  You can reference the property [this.seconds]. You can reference the function [this.ToString].  You can reference a non existing property / function like [this.Weird], but redirection will not work.

EXTERNAL REFERENCES

Let's suppose you are documenting the class [vehicleBaseObject].  You can reference another class property [VehicleComponent.damageLevel]. You can reference another class function [VehicleComponent.IsDriver]. You can reference an enum value [vehicleExitDirection.Front], it works with bitfields too.  You can reference a non existing function [VehicleComponent.Weird], but redirection will not work.

LINKS

You can add links to web sites. You can go to [Cyberpunk](https://www.cyberpunk.net/) for example. You could also learn some fundamentals about the [Entity Component System](https://en.wikipedia.org/wiki/Entity_component_system) pattern.
